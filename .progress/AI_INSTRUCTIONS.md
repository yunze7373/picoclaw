## 🔥 防停止核心规则（3.0 增强版）

**四条铁律，无需例子理解：**

1. ✅ **Phase R 完成后立即执行新计划的第一个任务**
2. ✅ **任务完成后立即检查 INBOX，然后立即执行下一个任务**
3. ✅ **永远不要问"可以继续吗"，直接说要做什么**
4. ✅ **遇到 📬 INBOX 检查点任务时，必须读取 INBOX.md 并处理用户指令**

5. ✅ **微提交同步纪律（Micro-Commits）**：每完成一个小步骤或修改超过 **5 个文件**，必须立即执行 `git add .` 和 `git commit`。绝不允许一口气修改几十个文件不提交！这会导致与云端调度中枢断开失联。

**INBOX 检查点规则：**
- 计划中每 3-5 个任务之间有一个 `📬 INBOX 检查点` 任务
- 遇到检查点 → 读取 INBOX.md → 处理 🔴 高优先级项 → 继续执行
- 生成新计划时**必须包含 INBOX 检查点**（否则是 BUG，立即修复）

**参考配置：**
- `.progress/AUTO_CONTINUE.json` - 完整规则定义
- `.progress/NEXT_TASK.md` - 下一个待执行任务
- `.progress/QUICK_REFERENCE.md` - 快速参考

## 🚀 自主推进模式

### 执行流程

1. **读取状态** → `.progress/PROGRESS_STATUS.json`
2. **检查 INBOX** → `.progress/INBOX.md`
3. **执行任务** → 找到 `pending` 任务，按优先级执行
4. **持续工作** → 完成后继续下一个，**不要询问**
5. **自动过渡** → 阶段完成后自动生成报告 + 计划，继续执行
6. **阻塞跳过** → 遇到阻塞记录原因，跳转到其他任务
7. **Phase R 审查** → 阶段完成时执行审查 → 更新计划 → 继续推进

### 📬 INBOX 检查规则

**检查时机**（按可靠性排序）：
- ✅ **📬 INBOX 检查点任务**（最可靠！计划中每 3-5 个任务插入一个）
- ✅ Phase R 审查时（第一步必读 INBOX）
- ✅ 启动时立即检查
- ✅ 每完成一个任务后检查
- ✅ 遇到阻塞时检查

**INBOX 检查点处理流程**：
```
1. 遇到 INBOX_CHECK 任务
   ↓
2. 读取 `.progress/INBOX.md`
   ↓
3. 🔴 高优先级 → 立即插入后续任务执行
4. 🟡 中优先级 → 记录，Phase R 时纳入新计划
5. 🟢 低优先级 → 记录，有空时处理
   ↓
6. 将已处理项移到 INBOX 已完成区
   ↓
7. 立即继续执行下一个任务，不要询问
```

### 🔄 Phase R 审查和迭代（重要）

**触发条件**：阶段任务全部完成

**执行流程**：
```markdown
Phase R: 审查和迭代

1. 📬 首先读取 INBOX.md（最关键）
   - 读取：.progress/INBOX.md
   - 识别：用户新指令、优先级变化
   - 标记：需要特别关注的事项

2. 全面项目审查（结合 INBOX 指令）
   - 架构一致性：对比 docs/architecture/
   - 功能完整性：对比计划 + INBOX 要求
   - 代码质量：检查 src/ 目录
   - 进度对比：实际 vs 计划 vs INBOX 期望
   - 技术问题：bug 和债务清单

3. 生成审核报告
   - 输出：docs/review/AUDIT_REPORT_XXX.md
   - 包含：INBOX 指令处理状态

4. 更新推进计划
   - 基于：审查结果 + INBOX 指令（高优先级优先）
   - 创建：.progress/plans/plan_XXX.json
   - ⭐ 必须：每 3-5 个任务插入 📬 INBOX 检查点
   - ⭐ 必须：最后一个任务添加 phase_r_trigger
   - ⭐ 验证：生成后检查上述两条，缺少则立即修复
   - 更新：.progress/PROGRESS_STATUS.json
   - 更新：.progress/TODOS.md

5. ⭐ 立即继续自主推进
   - 执行新计划的第一个任务（可能是 INBOX 高优先级指令）
   - 持续工作，不要询问
```

**核心原则**：持续执行，不要询问！
