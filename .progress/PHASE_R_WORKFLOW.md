# Phase R 审查和迭代流程

> 阶段完成后自动触发：审查 → 报告 → 更新计划 → 继续推进

---

```
阶段任务完成
        ↓
┌──────────────────────────────────────┐
│  Phase R: 审查和迭代                  │

```

**违反以上流程 = 系统故障**

---

## 核心原则

**Phase R 的最终目标不是生成报告，而是：**
1. 生成下阶段计划
2. **立即开始执行新计划**

```
Phase N 执行 → Phase N 完成 → Phase R 审查
                                    ↓
                            Phase N+1 计划生成
                                    ↓
                            ⭐ 立即执行 Phase N+1
                                    ↓
                            Phase N+1 完成 → Phase R 审查
                                    ↓
                            (循环继续)
```

## 一键启动审查

### 完整命令

```markdown
@workspace 阶段任务已完成，执行 Phase R 审查和迭代：

- 识别用户新指令
- 识别优先级变化
- 标记需要特别关注的事项

## R2. 🤖 Agents 审查流水线

**启动 phase_r_review pipeline（四个专家 Agent 并行审查）：**

| Agent | 审查维度 | 输出 |
|-------|----------|------|
| `staff-code-reviewer` | 代码正确性、安全、性能、可维护性 | findings + quality scores |
| `principal-architect` | 架构一致性、模块边界、数据流 | 架构偏离清单 |
| `software-test-engineer` | 测试覆盖、用例完整性 | 覆盖率报告 + 缺失测试清单 |
| `technical-documentation-engineer` | 文档完整性、API 文档 | 缺失文档清单 |

**审查流程：**

```
① 四个 Agent 并行执行审查
    ↓
② 合并审查结果 + INBOX 指令
    ↓
③ 按 P0/P1/P2 优先级分类发现
    ↓
④ 输入 R3 生成报告
```

**审查维度**（结合 INBOX 指令）：

1. **架构一致性**（principal-architect）
   - 对比文件：`docs/architecture/ARCHITECTURE.md`
   - 检查：实现是否符合设计的架构
   - 输出：符合点清单、偏离点清单

2. **代码质量**（staff-code-reviewer）
   - 检查范围：`src/` 目录
   - 检查：正确性、安全、性能、可维护性、可测性
   - 输出：quality scores + critical/major/minor issues

3. **测试覆盖**（software-test-engineer）
   - 检查范围：`tests/` 目录
   - 检查：覆盖率、缺失的测试用例、edge cases
   - 输出：覆盖率数据 + 建议补充的测试

4. **文档完整性**（technical-documentation-engineer）
   - 检查范围：`docs/`、README、API 文档、代码注释
   - 检查：缺失的文档、过时的文档
   - 输出：缺失文档清单 + 更新建议

5. **功能完整性**
   - 对比文件：`.progress/plans/plan_*.json`
   - 检查：计划的功能是否都实现了
   - 输出：完成功能清单、未完成功能清单

6. **参考项目借鉴**
   - 对比文件：`references/` 目录、`docs/analysis/*.md`
   - 检查：是否充分利用了参考项目
   - 输出：已借鉴清单、可借鉴但未利用清单

## R4. 更新推进计划

**任务**：
1. 提取审核报告中的改进建议
2. 结合 INBOX.md 中的新指令（高优先级优先）
3. 创建新任务计划 `.progress/plans/plan_002.json`
4. 更新状态文件 `.progress/PROGRESS_STATUS.json`
5. 更新任务清单 `.progress/TODOS.md`

**要求**：
- P0 必须完成 → 高优先级
- P1 应该完成 → 中优先级
- P2 可以完成 → 低优先级
- 每个任务有明确的 definition_of_done
- ⭐ **每个任务必须指定 agent 和 pipeline**（引用 agents-registry.json）
- ⭐ **每 3-5 个任务之间插入 📬 INBOX 检查点任务**
- ⭐ **最后一个任务必须包含 phase_r_trigger**
- ⭐ 生成后验证：agent分配 + 检查点 + phase_r_trigger 是否存在，缺少 = BUG → 立即修复

## R5. 继续自主推进

**指令**：
1. 读取新计划 `plan_002.json`
2. 执行第一个 P0 任务（可能是 INBOX 高优先级指令）
3. 完成后更新状态
4. 继续下一个任务
5. 持续工作，不要询问

---

**执行要求**：
- 每个阶段输出到对应文件
- 审查要具体，问题要可执行
- 不要询问，自主推进
- 审查后必须继续开发

开始！
```

## 最佳实践

### ✅ 推荐做法

1. **阶段完成必审查** - 不要直接继续
2. **首先读取 INBOX** - 确保响应用户最新指令
3. **审查要具体** - 问题要具体可执行
4. **优先级明确** - P0/P1/P2 分类
5. **审查后必行动** - 基于报告更新计划
6. **行动必继续** - 审查后立即执行新任务
7. **INBOX 优先** - 用户新指令优先处理
8. **参考借鉴审查** - 检查是否充分利用 references/
9. **新计划必含检查点** - 每 3-5 个任务插入 📬 INBOX 检查点
10. **新计划必含触发器** - 最后任务必须有 phase_r_trigger
11. **任务必分配 Agent** - 每个任务指定 agent + pipeline
12. **Phase R 用审查流水线** - 四个 Agent 并行审查，不要笼统检查

### ❌ 避免做法

1. ❌ 审查流于形式
2. ❌ 问题太模糊无法执行
3. ❌ 审查后不更新计划
4. ❌ 审查后停下询问
5. ❌ 同样的问题反复出现
6. ❌ 审查太频繁（影响进度）
7. ❌ 忽略参考项目（站在巨人肩膀上开发）

## 审查频率建议

| 项目阶段 | 审查频率 | 触发条件 |
|----------|----------|----------|
| Phase 0-1 | 每个 Phase 完成后 | 阶段目标完成 |
| Phase 2+ | 每 1-2 周 | Sprint 结束 |
| 上线前 | 每周 | 周五审查 |

---

## 关键要点

1. **Phase R 是必须阶段** - 不是可选
2. **首先读取 INBOX** - 确保响应用户最新指令
3. **审查是为了改进** - 不是批评
4. **改进要有行动** - 不只是报告
5. **行动要有结果** - 任务要完成
6. **结果要有验证** - 下次审查检查
7. **审查后继续执行** - 不要停下
8. **充分利用 references/** - 站在巨人肩膀上开发

---

> 审查和迭代 = 持续发现差距 → 持续改进 → 持续接近目标  
> 不是一次性完成，而是螺旋式上升！
