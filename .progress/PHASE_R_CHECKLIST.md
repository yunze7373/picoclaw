# Phase R 永动自检清单 (Phase R Checklist)

> **用途**：在引擎执行完一个阶段（Phase）准备进入 `Phase R` 时，**必须逐项核对本清单**，确保不会发生漏读 `INBOX`、漏生成下一计划、或漏切换到新任务的致命错误。只有全部核对打钩后，才允许继续。

---

## 一、触发前检查（开始前）

- [ ] 已预先读取 `.progress/INBOX.md` 了解最新插入的指令
- [ ] 已读取 `.progress/PROGRESS_STATUS.json` 确认当前节点
- [ ] 已读取刚刚执行完毕的计划文件（如 `.progress/plans/plan_034.json`）
- [ ] 已确认 INBOX 中新增的 🔴[P0] / 🟡[P1] 高优先级事项，并准备将其编入新计划
- [ ] 已确认上一阶段是否存在被阻塞项，并决定延期策略或修复策略

## 二、审查与清理阶段

- [ ] 已执行 **Memory Archiving (MAP)**：把 INBOX 和 TODOS 中带有 `[x]` 的历史项移入 `.progress/archive/`
- [ ] 已生成阶段性 **Git Commit**，永久保存当前 Phase 的成就，建立能够时光倒流的锚点
- [ ] 已检查代码实现成果与计划目标的一致性
- [ ] 已评估遗留的技术债、架构和测试覆盖率
- [ ] 已更新 INBOX 中刚被解决条目的状态（完成 / 延后 / 阻塞）
- [ ] 已正式输出本阶段结案报告，存入 `.progress/reports/` 或 `docs/review/` 

## 三、计划生成与交棒阶段

- [ ] 新计划文件（如 `.progress/plans/plan_035.json`）已准确创建
- [ ] **致命检查**：新计划 json 的最后一周**必须**包含 `phase_r_trigger` 流水线钩子
- [ ] `.progress/PROGRESS_STATUS.json` 已更新最新状态（状态为 in_progress，指针指向新计划）
- [ ] `.progress/TODOS.md` 已被格式化清空旧任务，更新为新计划任务序列，且顶部保留了轮转声明

## 四、收尾与点火（最关键）

- [ ] 已确认**绝对不会**以“输出结案语后停止”的方式结束
- [ ] 已明确新计划中第一个要干的任务是什么
- [ ] **立即开始执行新计划第一个任务，严禁停下来询问用户！**

---

## ❌ 快速失败信号 (如果出现，立刻纠正！)

如果你在自我反省中发现存在以下任一行为，说明你的 Phase R 执行严重错误：

- 没读 `INBOX.md` 就开始自顾自写新计划。
- 只生成了报告，忘了生成下一个阶段的 `.json` 计划。
- 生成了新计划，但是里面漏写了 `phase_r_trigger`。
- 生成了新计划，但忘了同步覆盖到 `TODOS.md`，导致你下一轮忘了自己要干嘛。
- **最致命错误**：完成审查后，你停在那问用户“我们下一步该怎么办？”或“是否继续？”。

---

## 推荐执行读写顺序（速查）

1. 读取 `INBOX.md`
2. 读取 `PROGRESS_STATUS.json`
3. 截断移动历史记录 (Archiving)
4. 生成 `AUDIT_REPORT_*.md`
5. 生成新 `plan_*.json`
6. 更新 `PROGRESS_STATUS.json` + `TODOS.md`
7. ✨ 立即自动动手执行新任务 ✨
